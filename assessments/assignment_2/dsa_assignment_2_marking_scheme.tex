\subsection*{A1. Worst-Case Time Complexity of \texttt{algorithm\_x} (4 Marks)}

\subsubsection*{Model Answer}
\texttt{algorithm\_x} contains two nested \texttt{for} loops. The outer loop runs $N$ iterations. For each iteration $i$, the inner loop runs from $i$ to $N-1$, executing approximately $N - i$ iterations. The total number of comparisons is:
\[
\sum_{i=0}^{N-1}(N - i) = N + (N-1) + \cdots + 1 = \frac{N(N+1)}{2} = O(N^2)
\]
The dominant cost is the nested comparison inside both loops.

\textbf{Worst-Case Time Complexity: $O(N^2)$} \\
\textbf{Space Complexity: $O(1)$} (no auxiliary data structure is used).

\subsection*{A5. Exponential Fibonacci and Memoisation (4 Marks)}

\subsubsection*{Model Answer}
Without caching, each call to $f(n)$ spawns two sub-calls: $f(n-1)$ and $f(n-2)$. This creates a binary recursion tree with $2^N$ nodes in the worst case because sub-problems are recomputed repeatedly (e.g., $f(3)$ is computed many times independently). The tree height is $N$ and at each level the number of nodes roughly doubles.

To reduce to $O(N)$ time: replace recursion with a simple iterative loop that tracks only the previous two values, eliminating all recomputation:
\begin{lstlisting}[language=Python]
def fibonacci(n):
    if n <= 1:
        return n
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b
\end{lstlisting}
This uses $O(1)$ space (only two variables) and $O(N)$ time (a single loop).

\begin{markbox}{A5 --- 4 Marks}
\begin{itemize}
    \item Correct explanation of the binary recursion tree causing repeated sub-problems: \textbf{2 Marks}
    \item Correct iterative solution achieving $O(N)$ time: \textbf{1 Mark}
    \item States $O(1)$ space and explains why (only two variables): \textbf{1 Mark}
\end{itemize}
\end{markbox}

\subsubsection*{Model Implementation}
\begin{lstlisting}[language=Python]
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
\end{lstlisting}

\begin{itemize}
    \item \textbf{Best Case: $O(1)$} --- the target is the middle element on the first comparison.
    \item \textbf{Worst Case: $O(\log N)$} --- the target is not present or is at an extreme end; the search space is halved $\log_2 N$ times until \texttt{left > right}.
\end{itemize}

def find_pair_with_sum(arr, target):
    left, right = 0, len(arr) - 1
    while left < right:
        current_sum = arr[left] + arr[right]
        if current_sum == target:
            return (arr[left], arr[right])

def rotate_array(arr, k):
    n = len(arr)
    k = k % n   # handle k > n gracefully

    def reverse(lo, hi):
        while lo < hi:
            arr[lo], arr[hi] = arr[hi], arr[lo]
            lo += 1
            hi -= 1

    reverse(0, n - 1)       # Step 1: reverse entire array
    reverse(0, k - 1)       # Step 2: reverse first k elements
    reverse(k, n - 1)       # Step 3: reverse remaining elements

# Demonstration:

\subsection*{B4. Amortised $O(1)$ Append (4 Marks)}

\subsubsection*{Model Answer}
When a Python list's internal capacity is full, a new array of approximately double the size is allocated and all existing elements are copied over. This single copy operation costs $O(N)$. However, this costly resize happens very rarely --- only after $N$ cheap appends. When the total cost is averaged (amortised) over all $N$ operations:
\[
\text{Amortised cost} = \frac{\text{Total work for } N \text{ appends}}{N} = \frac{O(N)}{N} = O(1)
\]
The key insight is that because the capacity doubles each time (geometric growth), the number of resize events is $O(\log N)$ over $N$ appends, making the cost per append effectively constant in the long run.

\begin{markbox}{B4 --- 4 Marks}
\begin{itemize}
    \item Correctly describes capacity doubling and the copy trigger: \textbf{1.5 Marks}
    \item Correct explanation of amortisation (spreading the one-time cost over many operations): \textbf{1.5 Marks}
    \item Notes that geometric growth ($\times 2$) is what makes the average constant: \textbf{1 Mark}
\end{itemize}
\end{markbox}

def cancel_booking(self, booking_id):
    current = self.head
    while current:
        if current.booking_id == booking_id:
            # Fix the backward link of the successor
            if current.next:
                current.next.prev = current.prev
            else:
                self.tail = current.prev    # deleted the tail

            # Fix the forward link of the predecessor
            if current.prev:
                current.prev.next = current.next
            else:
                self.head = current.next    # deleted the head

            return True   # found and deleted
        current = current.next
    return False           # not found
\end{lstlisting}

\subsection*{C4. \texttt{find\_and\_swap} (4 Marks)}

def find_and_swap(self, id1, id2):
    node1, node2 = None, None
    current = self.head
    while current:
        if current.booking_id == id1:
            node1 = current
        if current.booking_id == id2:
            node2 = current
        current = current.next

    if not node1 or not node2:
        return False   # one or both IDs not found

    # Swap data fields without touching any pointers
    node1.booking_id,   node2.booking_id   = node2.booking_id,   node1.booking_id
    node1.student_name, node2.student_name = node2.student_name, node1.student_name
    node1.destination,  node2.destination  = node2.destination,  node1.destination
    return True
\end{lstlisting}

\textbf{Time Complexity:} $O(N)$ --- a single traversal to locate both nodes. \\
\textbf{Why swap data?} Swapping data is simpler and less error-prone than relinking pointers, especially in a doubly linked list where both \texttt{prev} and \texttt{next} links must be updated for multiple surrounding nodes. Data swapping has no impact on the list's structural links.

\subsection*{C6. \texttt{BoardingQueue} and \texttt{deque} Justification (5 Marks)}

\subsubsection*{Model Implementation}
\begin{lstlisting}[language=Python]
from collections import deque

class BoardingQueue:
    def __init__(self):
        self._queue = deque()

    def join(self, student_name):
        """Student joins the back of the queue.  O(1)."""
        self._queue.append(student_name)

    def board(self):
        """Next student boards (leaves the front).  O(1)."""
        if not self._queue:
            return None
        return self._queue.popleft()

    def peek_next(self):
        """View next student without removing them.  O(1)."""
        if not self._queue:
            return None
        return self._queue[0]

    def size(self):
        """Number of students in the queue.  O(1)."""
        return len(self._queue)
\end{lstlisting}

\textbf{Why \texttt{deque} over \texttt{list} for a queue?} \\
Removing the first element from a Python \texttt{list} (\texttt{list.pop(0)}) takes $O(N)$ time because every remaining element must be shifted one position forward. A \texttt{collections.deque} is implemented as a doubly-ended linked structure that supports both \texttt{append} (right) and \texttt{popleft} (left) in $O(1)$ time, making it the correct choice for a FIFO queue.

\section{Part D: Hash Tables and Binary Search Trees --- Solutions (25 Marks)}

\subsection*{D1. Hash Collision Explanation (3 Marks)}

\subsubsection*{Model Answer}
A \textbf{hash collision} occurs when two different keys produce the same index after the hash function and modulo operation are applied (e.g., \texttt{hash("apple") \% 10 == hash("cherry") \% 10 == 7}). Since the underlying array has only one slot at each index, a strategy is needed to handle multiple keys competing for the same position.

\textbf{Resolution Strategies:}
\begin{enumerate}
    \item \textbf{Chaining (Open Hashing):} Each array slot holds a linked list of all key-value pairs whose keys hash to that index. Lookup walks the chain. If all $N$ keys hash to the same index (poor hash function), the chain degrades to a single linked list with $O(N)$ lookup.
    \item \textbf{Open Addressing (Linear Probing):} All entries are stored directly in the array. On collision at index $i$, the algorithm probes $i+1$, $i+2$, \ldots\ until an empty slot is found. Lookup follows the same probe sequence. In the worst case (table nearly full), this also degrades to $O(N)$.
\end{enumerate}

\begin{markbox}{D1 --- 3 Marks}
\begin{itemize}
    \item Correct definition of collision: \textbf{1 Mark}
    \item Correct description of Chaining with worst-case $O(N)$: \textbf{1 Mark}
    \item Correct description of Open Addressing / Linear Probing with worst-case $O(N)$: \textbf{1 Mark}
\end{itemize}
\end{markbox}

\textbf{Why does in-order traversal produce sorted output?} \\
By the BST property, every node in a node's left subtree has a smaller key, and every node in the right subtree has a larger key. Visiting left subtree $\rightarrow$ root $\rightarrow$ right subtree recursively enforces this ordering at every level, producing a globally ascending sequence.

\subsection*{D6. \texttt{search} and \texttt{find\_range} (5 Marks)}

\subsubsection*{Model Implementation}
\begin{lstlisting}[language=Python]
def search(root, grade_score):
    if root is None:
        return None                       # not found
    if grade_score == root.grade_score:
        return root.student_id            # found
    elif grade_score < root.grade_score:
        return search(root.left, grade_score)
    else:
        return search(root.right, grade_score)


def find_range(root, low, high):
    results = []
    def _dfs(node):
        if node is None:
            return
        if low < node.grade_score:        # prune: left subtree may have values >= low
            _dfs(node.left)
        if low <= node.grade_score <= high:
            results.append(node.student_id)
        if node.grade_score < high:       # prune: right subtree may have values <= high
            _dfs(node.right)
    _dfs(root)
    return results                        # already in ascending order via in-order pruning
\end{lstlisting}

\textbf{Time Complexities:}
\begin{itemize}
    \item \texttt{search}: $O(H)$ where $H$ is the height of the BST. In the best case (balanced tree), $H = O(\log N)$. In the worst case (completely skewed / sorted insertion), $H = O(N)$.
    \item \texttt{find\_range}: $O(H + K)$ where $K$ is the number of nodes within the range. The pruning avoids visiting subtrees that cannot contain values in $[low, high]$.
\end{itemize}

\textbf{When $H = O(\log N)$ vs $H = O(N)$:}
\begin{itemize}
    \item $H = O(\log N)$: The tree is balanced --- nodes are distributed evenly between left and right subtrees at each level (e.g., AVL tree or random insertion order).
    \item $H = O(N)$: The tree is fully skewed --- elements were inserted in already-sorted (ascending or descending) order, producing a straight chain equivalent to a linked list.
\end{itemize}

\begin{itemize}
    \item Correct recursive \texttt{search} with three-way branch and \texttt{None} base case: \textbf{2 Marks}
    \item Correct \texttt{find\_range} with BST pruning (avoids visiting out-of-range subtrees): \textbf{2 Marks}
    \item Correct explanation of $O(\log N)$ vs $O(N)$ height with concrete examples: \textbf{1 Mark}
\end{itemize}

def build_conflict_map(sessions):

    room_to_indices = {}

        if room_id not in room_to_indices:

def detect_first_conflict(room_sessions):
    """
    room_sessions: list of (index, start, end) tuples, sorted by start time.
    Returns the first conflicting pair of indices, or None.
    """
    stack = []   # each entry is (index, start, end)
    for entry in room_sessions:
        idx, start, end = entry
        if stack and start < stack[-1][2]:   # start of current < end of top
            return (stack[-1][0], idx)        # conflict detected
        stack.append(entry)
    return None
\end{lstlisting}

\textbf{How the Stack is used:} The stack always holds sessions in progress (those that have started but not yet ended). When a new session starts before the top-of-stack session ends, a conflict is detected immediately and the pair is returned.

\subsection*{E3. Complexity Analysis and Alternative Approaches (3 Marks)}

\subsubsection*{Model Answer}
\textbf{Complexity of the E1 solution:}
\begin{itemize}
    \item \textbf{Grouping step:} $O(N)$ --- a single pass over all sessions.
    \item \textbf{Pairwise comparison per room:} $O(M^2)$ for a room with $M$ sessions, $O(N^2)$ in the worst case if all sessions are in one room.
    \item \textbf{Space Complexity:} $O(N)$ for the grouping hash table and results.
\end{itemize}

\textbf{Would a BST or sorted array with binary search improve performance?} \\
Yes. If sessions within each room are sorted by \texttt{start\_time}, we can use the following approach:
\begin{itemize}
    \item Sort the sessions per room by \texttt{start\_time}: $O(M \log M)$.
    \item After sorting, a single linear scan is sufficient: maintain only the maximum \texttt{end\_time} seen so far. If the next session's \texttt{start\_time} is less than the running maximum end time, a conflict exists. This reduces pairwise detection from $O(M^2)$ to $O(M)$ after the $O(M \log M)$ sort, yielding an overall $O(N \log N)$ solution.
    \item A BST keyed on \texttt{start\_time} can achieve the same after $O(M \log M)$ insertions, but offers no asymptotic advantage over a sorted array here since we still need a linear scan of the sorted order.
\end{itemize}

    \item Correct description of the sort + linear scan improvement ($O(N \log N)$): \textbf{1.5 Marks}
    \item Honest comparison of BST vs sorted array for this problem: \textbf{0.5 Marks}
